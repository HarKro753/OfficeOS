
## Brand System

### Name

**OfficeOS**

OfficeOS sounds like an operating system, which is useful. The name should be treated as the system of record and operating layer for app delivery work.

Do not over-explain the "Office" part. Let the OS meaning carry the brand.

### Positioning

**OfficeOS is the app delivery OS for teams that know what app they want but do not want to run the mobile production lifecycle themselves.**

### Taglines

Primary:

**Bring the app direction. OfficeOS owns delivery.**

Alternatives:

- From app idea to operating product.
- The operating layer after the app handoff.
- Accepted spec in. Shipped app out.
- App delivery, release, and upkeep in one accountable workflow.

### Logo Concept

Use a wordmark plus a modular OS mark.

**Mark idea:** A 3x3 isometric cube grid with one cube highlighted as the active build. The grid represents product, design, readiness, implementation, QA, release, analytics, paywall, and monitoring. The highlighted cube shows that OfficeOS turns scattered lifecycle work into an organized operating system.

**Logo style:**

- Geometric
- Sharp but not aggressive
- Monoline or solid modular construction
- Works in one color
- Works as a small app icon

**Avoid:**

- Generic robot head
- Sparkles
- Cloud icon alone
- Office building icon
- Abstract AI swirl

### Color Palette

OfficeOS should avoid the default purple AI SaaS look. Use a crisp operational palette with one energetic accent.

|Role|Color|Hex|Use|
|-|-|-|-|
|Command Black|Near-black|`#101418`|Primary text, logo, dark panels|
|System White|White|`#FFFFFF`|Main background|
|Blueprint|Deep blue|`#2457FF`|Primary action, active pipeline, selected states|
|Signal Green|Green|`#20B26B`|Accepted, passed, live monitoring|
|Gate Amber|Amber|`#F5A524`|Needs review, clarification, warning|
|Release Red|Red|`#E5484D`|Failed checks, rejected builds, critical issues|
|Slate 700|Muted text|`#46515D`|Secondary text|
|Slate 100|Light surface|`#EEF2F5`|Panels, table backgrounds, inactive modules|
|Line Gray|Border|`#D8DEE4`|Dividers, diagrams, lanes|

**Palette rule:**

White and near-black carry the interface. Blue shows active ownership. Green, amber, and red are operational states, not decoration.

### Typography

Use a modern grotesk for clarity and a monospace for source-of-truth artifacts.

|Use|Font direction|Notes|
|-|-|-|
|Headlines|Inter, Geist, or Söhne-style grotesk|Compact, confident, not playful|
|Body|Inter or Geist Sans|Readable, neutral, operator-like|
|Code/docs|JetBrains Mono or Geist Mono|Use for `SPEC.md`, `DESIGN.md`, `ChangeRequest.md`, status labels|

**Type rule:**

Headlines should be short and declarative. Do not use soft startup phrasing.

Good:

- OfficeOS owns the app delivery stack.
- The source of truth drives every build.
- The project is over. The lifecycle just started.

Weak:

- Build apps effortlessly with AI.
- The future of mobile is here.
- Your all-in-one innovation partner.

### Graphic Language

Use systems, not decoration.

**Core assets:**

- Lifecycle pipeline
- Readiness gate
- Source-of-truth document cards
- Mobile build preview
- Module cube stack
- Status rail
- Change request diff

**Shape rules:**

- 8px radius for product UI panels
- 4px radius for small labels and status pills
- Thin 1px borders for structure
- Avoid very rounded bubbly SaaS cards
- Avoid decorative gradients as the main visual identity

**Isometric module style:**

Use light-gray isometric cubes with blue active faces and green passed-check highlights. Keep the cubes clean and diagrammatic, not playful.

### Voice

OfficeOS should sound accountable, direct, and operational.

**Voice attributes:**

- Clear
- Specific
- Calm
- Ownership-oriented
- Practical

**Say:**

- accepted source of truth
- readiness gate
- delivery lifecycle
- release preparation
- operating layer
- maintenance workflow
- app updates
- monitoring active

**Avoid:**

- magic
- autonomous everything
- effortless
- instant app empire
- revolutionizing software
- vibe-coded apps in seconds

### Slide Design Rules

The hackathon deck should feel like a product operating system, not a marketing landing page.

**Deck rules:**

- One concrete claim per slide.
- Big visual system diagram on every core slide.
- Keep text short enough that the speaker carries the story.
- Use status language in visuals so judges see the workflow working.
- Repeat the same pipeline metaphor across problem, solution, demo, and business model.

**Slide visual recipe:**

- Background: `#FFFFFF`
- Text: `#101418`
- Primary accent: `#2457FF`
- State accents: green, amber, red only when the slide has operational status
- Diagram lines: `#D8DEE4`
- Module surfaces: `#EEF2F5`

### UI Direction

OfficeOS product UI should feel like Linear, Vercel, and a mobile release dashboard had a restrained enterprise child.

**Primary screens:**

- App intake
- Source-of-truth review
- Readiness gate report
- Build pipeline
- Release checklist
- Monitoring dashboard
- Change request queue

**Interface principles:**

- Dense but readable
- Status-first
- Clear ownership by lane
- Documents and builds are first-class objects
- Every workflow has an explicit next action

**UI components:**

- Left navigation rail
- App status table
- Readiness score panel
- Build timeline
- Diff viewer for `SPEC.md` and `DESIGN.md`
- Module toggles for analytics, paywall, monitoring, release support
- Operator notes panel
